### 三、技术领域

本发明涉及自动驾驶远程运营（Remote Operations）技术领域，特别涉及一种基于三维场景映射表、结构化异常事件分发和动态层级重分配的L4级无人驾驶出租车异常事件处置方法及系统。

### 四、背景技术

#### 4.1 现有技术

**（1）Waymo Fleet Response咨询模式**

Waymo采用顾问模式（Advisory Model）的远程运营：AI始终保持车辆完整控制权，远程操作员仅在AI请求时提供情境信息辅助。异常分发基于AI单次请求触发，由操作员手动分配。该方案的局限在于：事件分发依赖操作员主观判断，缺乏结构化分类；无法处理系统性批量故障（2025年12月旧金山断电事件期间，Waymo无法及时召回被困车辆）。

**（2）Tesla Teleops直接控制模式**

Tesla的远程操作方案允许操作员在紧急情况下直接接管车辆控制（≤10mph）。异常事件通过规则引擎分类：若车辆停止时间超过阈值则升级到远程接管，否则继续等待AI决策。规则引擎方案的局限在于：规则数量有限且手工编写，难以覆盖所有异常类型；无法对同一场景按严重度/紧急度差异化处置。

**（3）百度Apollo萝卜快跑事件驱动响应**

百度Apollo远程运营系统基于事件触发：当车辆报告异常时，系统根据预设的事件类型编码（车辆故障/通信故障/外部干扰等）分配给对应技术员处理。该方案的局限性在2026年3月武汉大规模瘫痪事件中充分暴露：近百辆车同时熄火，但系统缺乏批量事件聚合分发能力；SOS按钮无人应答；最终依靠交警逐车施救。

**（4）学术界场景分类体系**

ISO 34502:2022提出自动驾驶测试场景分类方法，从功能场景→逻辑场景→具体场景三层递进。然而该分类针对测试场景（设计阶段），不适用于运营期异常事件分类。学术界缺乏专门针对Robotaxi运营异常的结构化分类体系。

#### 4.2 现有技术的不足

1. **异常分发缺乏结构化分类体系**：现有运营商使用各自自定义的事件编码（Waymo有内部编码、Tesla使用不同的规则），行业缺乏统一的异常场景分类基线，导致跨运营商的数据无法互通、监管难以横向对比。

2. **事件严重度与紧急度维度混淆**：现有方案常将"严重度"和"紧急度"混为一谈。例如S3大规模瘫痪事件（严重度高）可能紧急度也高，但S0未遂事件（严重度低）在乘客恐慌时紧急度也可能很高。混淆处理会导致资源配置失衡。

3. **规则引擎难以扩展和维护**：当新的异常场景出现时（如新城市新道路特性），规则引擎需要人工重新编写规则并部署，维护成本高，可追溯性差。

4. **缺乏可审计的处置日志**：黑盒调度系统难以在事后回溯"为什么这起事件被分配到这个层级"，不满足中国GB标准附录D安全档案要求的可审计性。

5. **无法实现操作员负载下的动态资源重分配**：现有方案的层级分配是静态的——一旦分配到L2（AI辅助+人工确认），无论操作员是否超负荷都不能再调整。实际运营中，当L2操作员超负荷时，本应降级到AI自主的事件被挤占处理资源。

6. **缺乏针对Robotaxi运营物理特性的分类维度**：现有分类（如事件类型、故障类型）是通用IT运维分类，未反映Robotaxi运营的物理本质——即"系统性故障→感知失效→规划异常→硬件失效→外部冲突→乘客端异常"这六类根本成因。

#### 4.3 本发明要解决的技术问题

- 问题P1：如何建立统一的、反映Robotaxi运营物理本质的异常场景分类体系？（对应不足1和6）
- 问题P2：如何将场景类型、严重度、紧急度三维度正交化，实现差异化分发？（对应不足2）
- 问题P3：如何实现异常分发的可审计性和可追溯性？（对应不足3和4）
- 问题P4：如何实现操作员负载感知的动态层级重分配？（对应不足5）

### 五、发明内容

#### 5.1 技术方案概述

本发明提出一种基于场景-处置映射表的L4无人驾驶出租车三层决策方法，包括以下步骤：

- **S1**（事件要素提取与三维特征化）：从ADS上报的异常事件中提取场景类型（6大类27子场景）、严重度（S0-S4）、紧急度（U0-U3）三维要素，建立事件的三维特征向量表征。
- **S2**（场景-层级映射表查询与事件分发）：基于预构建的三维映射表[scenario_code × severity × urgency] → {L1, L2, L3}，查询事件的初始分发层级；映射表由Robotaxi运营物理特性决定，不可简单迁移到其他领域。
- **S3**（BERT-based场景分类推理）：当事件以自由文本描述而非结构化编码上报时，使用专门微调的BERT分类器将文本映射到27个子场景编码之一，实现异构输入的统一处理。
- **S4**（操作员负载反馈的动态重分配）：基于各层级操作员当前负载（排队事件数、已处理事件数）实时调整分发决策，高负载时将部分L2降级到L1或L3，低负载时提升分发精度。
- **S5**（可审计的分发日志）：每次分发决策均记录"事件特征向量→映射表查询→分发结果"的完整推理链，满足GB标准附录D安全档案的可审计性要求。

#### 5.2 步骤S2：三维场景-层级映射表设计（核心创造性）

**（a）映射表的数学结构**

映射表M定义为从三维特征空间到决策层级的函数：

```
M: S × V × U → {L1, L2, L3}
```

其中：
- S = {A1, A2, ..., F4}：27个子场景编码（6大类分级）
- V = {S0, S1, S2, S3, S4}：5级严重度
- U = {U0, U1, U2, U3}：4级紧急度

映射表空间为27×5×4 = 540个三元组，但实际映射不是任意的，而是由Robotaxi运营的物理特性决定（见下文c项）。

**（b）六大类场景的物理成因分类**

与通用IT运维分类不同，本映射表的场景分类由Robotaxi运营的物理成因决定：

| 大类 | 物理成因 | 子场景数 |
|------|---------|---------|
| A类 系统性故障 | 云端/网络/电力基础设施级失效 | 4 |
| B类 感知/决策失效 | 车端传感器/AI决策模块异常 | 5 |
| C类 规划/执行异常 | 路径规划/控制执行错误 | 5 |
| D类 车辆硬件故障 | 物理部件（传感器/动力/制动）故障 | 4 |
| E类 外部环境冲突 | 其他交通参与者/道路/应急车辆 | 4 |
| F类 乘客端异常 | 乘客相关事件 | 4 |

**反"换应用场景"测试：** 如果将此分类应用到医疗机器人或工业机器人，则A/B/C/D分类仍可能适用（因其反映通用自主系统的共性故障模式），但E类"外部环境冲突"中的"交警手势理解失败"、F类"乘客在危险位置上下车"等子场景是Robotaxi运营特有的物理场景，无法迁移。因此本分类体系的27子场景级别与Robotaxi物理运营深度耦合。

**（c）映射表的具体内容（部分示例）**

```
M(A1, S3, U3) = L1(立即执行MRC) + L3(车队级恢复)  // 系统性云端故障+大规模瘫痪+乘客被困
M(A1, S1, U0) = L1(边缘决策恢复)  // 云端短暂中断+轻微+无紧急
M(B1, S0, U0) = L1(AI自动等待gap)  // 路口犹豫+未遂+无紧急
M(B1, S2, U2) = L2(人工确认路径)  // 路口犹豫+中等+高紧急
M(E3, S3, U3) = L2立即升级→L1让行  // 阻碍救护车+严重+立即
M(D3, S4, U3) = L3(紧急救援+119)  // 车辆起火+致命+立即
M(F1, S2, U3) = L1解锁+L3救援  // 乘客被困+中等+立即
```

**（d）映射表的存储与查询实现**

映射表存储为JSON格式，查询复杂度O(1)：

```python
mapping_table = {
    "A1_S3_U3": ["L1_MRC_IMMEDIATE", "L3_FLEET_RECOVERY"],
    "A1_S1_U0": ["L1_EDGE_RECOVERY"],
    ...
}

def dispatch(scenario_code, severity, urgency):
    key = f"{scenario_code}_{severity}_{urgency}"
    return mapping_table.get(key, ["L2_DEFAULT_HUMAN_REVIEW"])
```

**创造性论证：** 相比规则引擎（IF-THEN规则数量随场景线性增长，难维护）和神经网络黑盒分发（缺可审计性），本三维映射表实现了可解释+可扩展+可审计的三维决策分发。27子场景的分类维度是由Robotaxi运营物理特性决定的（非通用任务分类），且映射值由运营数据驱动迭代更新，构成与技术问题深度耦合的分类体系。

#### 5.3 步骤S3：BERT-based场景分类推理

**技术问题：** 实际运营中，车载ADS上报的事件可能是结构化编码（如"scenario=B1"），也可能是自由文本（如"车辆在路口犹豫30秒未能通过"）。需要统一映射到27子场景编码。

**（a）分类器架构**

使用预训练BERT模型（bert-base-chinese/bert-base-multilingual）+ 场景分类头（27-class softmax）：

```
输入文本 → BERT编码器 → [CLS]向量 → Dense(256) → Softmax(27) → 场景编码
```

**（b）训练数据**

训练集来源：
- ROAM事件数据库的description字段（已有16+事件标注）
- 各运营商公开事故报告（Waymo SGO报告、NHTSA报告）
- 合成数据增强（使用LLM生成相似描述）

训练数据规模目标：每子场景≥50条，共1350+条。

**（c）推理流程**

1. 事件上报文本预处理（分词、长度归一化至512）
2. BERT编码
3. 分类头输出27维概率分布
4. 选择最高概率的场景编码
5. 若最高概率<0.7，标记为"需人工审核"并降级到L2默认层

**（d）与通用文本分类的区别**

通用BERT文本分类器训练于新闻、情感等语料，无法区分"车辆在路口犹豫30秒"（B1 路口犹豫）与"车辆在路口停滞不动"（C1 路中停车/冻结）的细微差别——这两个子场景的处置层级不同（B1→L2，C1→L1）。本专利中的BERT微调使用Robotaxi特定语料，模型输出与27场景编码深度绑定。

#### 5.4 步骤S4：操作员负载反馈的动态层级重分配（核心创造性）

**技术问题：** 静态映射表可能将大量事件分发到L2，但L2操作员数量有限，超负荷时响应时间急剧增加。

**（a）操作员负载指标**

定义L层操作员负载指标Load(L,t)：

```
Load(L, t) = α·Q(L,t) + β·P(L,t) + γ·ART(L,t)
```

其中：
- Q(L,t)：时刻t在L层的排队事件数
- P(L,t)：时刻t在L层正在处理的事件数（每操作员可并发处理数，通常L2为1-3）
- ART(L,t)：最近15分钟的平均响应时间
- α=0.4, β=0.4, γ=0.2为权重

**（b）动态重分配规则**

当Load(L2,t) > 阈值θ_high（如0.85）时：

```
For each queued event e in L2:
  if e.severity <= S1 and e.urgency <= U1:
    reassign(e, L1)  # 降级到AI自主（使用保守策略）
  elif e.severity >= S3 or e.urgency >= U3:
    reassign(e, L3)  # 升级到遥控驾驶（如L3有容量）
```

当Load(L2,t) < 阈值θ_low（如0.4）时：
```
For each L1 event e with confidence < 0.85:
  upgrade(e, L2)  # 提升处理精度，人工复核
```

**（c）防抖动机制**

为避免事件在L1和L2之间频繁切换，引入滞后窗口：
- 重分配后，该事件至少在新层级停留10分钟
- 同一事件累计重分配次数≤2

**（d）与通用负载均衡的区别**

通用云计算负载均衡（如轮询、一致性哈希）不考虑事件的安全关键性和严重度/紧急度维度。本方案的重分配决策必须满足约束："降级不能违反安全底线"（如S4严重度事件不能降级到L1），这是Robotaxi运营特有的物理约束。

**1. 反"换应用场景"测试：**

**2. 反"常规组合"测试：**
