### 四、背景技术

#### 4.1 现有技术

**（1）Waymo Fleet Response基于4G/5G蜂窝网络的咨询模式**

Waymo的远程运营使用商业4G/5G蜂窝网络传输视频和控制信号。该方案的优势是部署快、成本低；局限在于：（1）蜂窝网络是"尽力而为"服务，无延迟保证；（2）实测端到端延迟150-500ms，在复杂电磁环境下可能波动至1秒以上；（3）Waymo为降低成本将部分运营中心迁至菲律宾，跨国延迟进一步增加至250ms+；（4）2025年12月旧金山断电期间，蜂窝网络同时受影响，Waymo无法及时召回被困车辆。

**（2）Tesla Teleops直接控制方案**

Tesla的远程操作允许操作员直接接管车辆控制，但速度限制在10mph以下。这一速度限制本质上是因为通信延迟无法保证：蜂窝网络的不确定延迟使得高速远程驾驶不安全。该方案的局限在于：（1）10mph限速严重影响运营效率；（2）单一蜂窝链路，无冗余；（3）操作员工作站使用普通PC+游戏手柄，缺乏力反馈。

**（3）Vay Technology低速远程驾驶**

德国初创公司Vay在拉斯维加斯运营远程驾驶共享车服务（17,000+次出行），采用专用4G+5G双网络冗余，端到端延迟约200ms。局限：仍为商业蜂窝网络，非物理层专用链路；速度限制在30mph。

**（4）DJI OcuSync无人机图传技术**

DJI OcuSync（O3/O4版本）是消费级和工业级无人机广泛采用的图传技术，技术参数：端到端延迟低至15ms（O4版本）、传输距离15km、自适应跳频覆盖5.170-5.250 GHz和5.725-5.850 GHz双频段、支持8架无人机同时操作。然而该技术目前仅用于无人机遥控，未在Robotaxi远程运营领域应用。

**（5）学术界人类神经肌肉延迟研究**

Cummings等（2026）系统总结美军35年UAV远程遥控经验，明确指出：人类神经肌肉延迟为200-500ms，要求操作员在200ms内响应的场景物理上不可行。然而学术研究未给出针对Robotaxi三层决策架构的差异化延迟预算设计方案。

#### 4.2 现有技术的不足

1. **单一蜂窝链路无法保证延迟确定性**：蜂窝网络是尽力而为服务，无QoS保证。Waymo/Tesla的方案在城市高密度区域、地下停车场、隧道等场景会出现延迟剧增。

2. **延迟预算未与决策层级匹配**：现有方案对所有远程操作使用相同的延迟预算（如<300ms），未区分AI辅助确认（L2）与远程直接驾驶（L3）的不同需求。L2只需确认AI方案，对延迟不敏感；L3需要实时操控，对延迟极敏感。

3. **缺乏通信链路的三维质量监测**：现有方案通常只监测单一指标（如RTT延迟），忽略了信号强度、延迟抖动、丢包率等影响控制安全的关键维度。

4. **缺乏链路质量不达标时的自动降级机制**：当蜂窝网络质量恶化时，现有系统要么继续使用（造成危险），要么报警请求人工干预（延迟响应）。缺乏自动降级到安全停车（MRC）的机制。

5. **工作站人机交互未吸收UAV领域经验**：现有Robotaxi远程工作站多使用普通PC+游戏手柄，未吸收UAV领域40年来的成熟方案（三段式UI、力反馈控制器、物理急停按钮）。

6. **物理层专用通信链路未被引入Robotaxi领域**：DJI OcuSync的物理层直连技术消除TCP/IP握手开销，可实现15ms延迟，但尚未被Robotaxi行业采纳。

7. **跨国运营中心增加不必要延迟**：Waymo菲律宾运营中心增加~100ms跨国延迟，对L3远程驾驶不可接受。

#### 4.3 本发明要解决的技术问题

- 问题P1：如何为不同决策层级分配差异化延迟预算？（对应不足2）
- 问题P2：如何引入物理层专用通信链路消除蜂窝网络不确定性？（对应不足1和6）
- 问题P3：如何实现链路质量恶化时的自动安全降级？（对应不足3和4）
- 问题P4：如何设计符合人因工程的远程操作员工作站？（对应不足5）

### 五、发明内容

#### 5.1 技术方案概述

本发明提出一种基于决策层级延迟预算的Robotaxi远程运营通信链路自适应选择方法及工作站系统，包括：

- **S1**（延迟预算分层设计）：按三层决策架构分配差异化延迟预算——L1（AI自主）无网络依赖，L2（AI辅助+人工确认）<500ms，L3（远程驾驶）<150ms。
- **S2**（通信链路自适应选择）：L1使用边缘本地通信，L2使用商业蜂窝网络（双载波冗余），L3使用借鉴DJI OcuSync架构的物理层专用图传链路（双频段5.17-5.25GHz+5.725-5.85GHz自适应跳频）。
- **S3**（链路质量三维实时监测）：同时监测信号强度RSSI、延迟抖动Jitter、丢包率Loss Rate三维指标，计算综合链路质量评分。
- **S4**（链路恶化自动降级决策）：当L3链路延迟超过150ms阈值或质量评分<0.6时，自动从远程驾驶降级到MRC（Minimum Risk Condition，靠边停车）。
- **S5**（三段式远程操作员工作站）：借鉴DJI FlightHub 2 Virtual Cockpit架构，实现"地图+遥测/FPV+控制面板"三段式UI，配备力反馈控制器（替代游戏手柄）和物理急停按钮。

#### 5.2 步骤S1：延迟分级预算设计（核心创造性）

**（a）人类神经肌肉延迟约束**

基于Cummings（2026）研究，人类响应外部刺激的神经肌肉延迟为200-500ms。这是物理约束，无法通过训练或技术缩短。

**（b）三层决策的延迟预算**

| 层级 | 决策类型 | 延迟预算 | 理论依据 |
|------|---------|---------|---------|
| L1 | AI自主响应 | N/A（边缘决策） | 不依赖远程通信 |
| L2 | AI辅助+人工确认 | <500ms | 人工确认允许人类神经肌肉延迟上限 |
| L3 | 远程驾驶 | <150ms | 需保证人类+通信+执行的总延迟<350ms(人类下限) |

**（c）延迟预算的数学推导**

对于L3远程驾驶：
```
总反应时间 = 通信延迟 + 人类反应延迟 + 车辆执行延迟
```

安全约束：总反应时间 < T_safe_action（约350ms，车辆在10km/h行驶3米内可停车）

分解：
- 人类反应延迟 = 200ms（最优状态下限）
- 车辆执行延迟 = 50ms（执行器响应）
- 剩余给通信延迟 = 100ms

考虑延迟抖动，设置通信延迟预算=150ms（含30%裕量）。

**（d）延迟预算的物理不可替换性**

**反"换应用场景"测试：** 150ms和500ms的数值来自交通安全的具体物理约束（车辆速度10km/h下3米内停车）。如果应用于无人机遥控（高度5米，速度30km/h），需要重新计算延迟预算。如果应用于机械臂遥控（速度1cm/s），延迟预算可放宽到1秒以上。因此本专利的延迟分级数值与Robotaxi物理特性深度耦合。

#### 5.4 步骤S3&S4：链路质量监测与自动降级

**（a）三维质量指标**

```python
def compute_link_quality(rssi, jitter, loss_rate):
    # RSSI评分 (dBm → 0-1)
    rssi_score = max(0, min(1, (rssi + 110) / 40))  # -110dBm=0, -70dBm=1

    # Jitter评分 (ms → 0-1)
    jitter_score = max(0, 1 - jitter / 50)  # 0ms=1, 50ms=0

    # 丢包率评分 (% → 0-1)
    loss_score = max(0, 1 - loss_rate / 5)  # 0%=1, 5%=0

    # 综合评分
    quality = 0.4 * rssi_score + 0.4 * jitter_score + 0.2 * loss_score
    return quality
```

**（b）自动降级决策逻辑**

```python
def link_degradation_check(layer, current_latency, quality):
    budgets = {'L1': float('inf'), 'L2': 500, 'L3': 150}

    if current_latency > budgets[layer] or quality < 0.6:
        if layer == 'L3':
            # L3链路不达标，立即降级至MRC
            return 'DOWNGRADE_TO_MRC'
        elif layer == 'L2':
            # L2链路不达标，切换至边缘决策
            return 'FALLBACK_TO_EDGE'
    return 'MAINTAIN'
```

**（c）降级执行流程**

1. 检测到L3链路质量<0.6或延迟>150ms
2. 车端立即启动MRC预案（靠边停车）
3. 向远程操作员发送降级通知
4. 乘客端广播语音提示
5. 记录降级事件到日志
