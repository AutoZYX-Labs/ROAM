## 第一章 软件概述

### 1.1 软件名称与版本

本软件中文名称为"ROAM无人驾驶出租车远程运营异常管理开源平台"，英文名称为"ROAM — RoboTaxi Operations Anomaly Management Platform"，简称"ROAM"。当前版本为V1.0，完成于2026年4月5日，首次公开发布于2026年4月2日。

### 1.2 开发背景与目的

L4级及以上无人驾驶出租车（Robotaxi）正在全球范围内加速商业部署。2025年第四季度，百度Apollo旗下"萝卜快跑"完成超过340万订单；Waymo在美国旧金山、凤凰城等城市运营超过3000辆无人车；Tesla、Pony.ai、WeRide等运营商也在快速扩展。

然而，当前Robotaxi远程运营存在严重的行业碎片化问题：

- **2026年3月31日，武汉"萝卜快跑"大规模瘫痪**：近百辆车同时熄火在高架路段，乘客被困2小时，SOS按钮无人应答，最终依靠交警逐车施救。
- **2025年12月，Waymo旧金山断电事件**：区域性停电导致Waymo车队大面积瘫痪，应急服务拨打Waymo热线31次，累计等待2.5小时。
- **2023年10月，Cruise行人拖拽事件**：远程操作员未能及时有效干预，导致Cruise被美国NHTSA罚款150万美元并全面召回车队。

这些事件揭示了一个共同问题：每家运营商都在建设自己独立的远程运营系统，缺乏统一的异常分类、处置参考和评价基准。

本软件的开发目的是：**为整个Robotaxi行业提供一个开源的远程运营异常管理基础平台**，降低各方建设远程运营系统的门槛，促进行业共识的形成。

### 1.3 主要功能概述

ROAM平台提供四个核心功能模块：

1. **事件数据库**：以结构化YAML格式记录全球Robotaxi运营异常事件，当前已收录16起典型事件，覆盖Waymo、Cruise、百度Apollo、Pony.ai四家主要运营商，时间跨度2023-2026年。

2. **场景分类体系**：建立6大类27子场景的异常事件分类框架，配合严重度（S0-S4）和紧急度（U0-U3）分级，实现事件的标准化归类。

3. **三层参考架构**：提出"AI自主响应→AI辅助+人工确认→远程驾驶/线下派遣"的三层决策模型，每层对应不同的响应时间、人员要求和处置方式。

4. **评价基准**：定义8项关键性能指标（KPI），包括平均处置时间（MTTR）、AI自主处置率、误升级率、乘客被困时长等，用于评估远程运营质量。

### 1.4 软件特点与创新点

本软件具有以下创新特点：

**特点一：开源共建模式**
软件采用Apache License 2.0开源许可证，在GitHub（https://github.com/AutoZYX/ROAM）公开发布，支持全球研究者、运营商和监管机构共同贡献事件数据、优化分类体系、改进参考架构。

**特点二：标准化数据格式**
所有事件数据采用统一的YAML Schema格式，字段包括事件ID、日期、运营商、地点、场景编码、严重度、紧急度、描述、影响、应急响应、根因、系统性问题、来源链接等。每起事件必须附带可验证的公开来源链接。

**特点三：与国际国内标准对齐**
软件的场景分类体系与ISO 34502（自动驾驶系统测试场景）、中国GB《智能网联汽车 自动驾驶系统安全要求》附录C.2（远程协助）、联合国WP.29 GTR（自动驾驶系统全球技术法规）等重要标准形成系统性对齐。

**特点四：跨域经验融合**
软件吸收了美军35年无人机（UAV）远程运营经验（Cummings, 2026）和DJI 30年工业无人机产品化经验（OcuSync/FlightHub 2/UTC认证），将跨域经验映射到Robotaxi三层架构的工程设计。

**特点五：纯静态部署**
软件的Web界面采用纯HTML/CSS/JavaScript实现，通过GitHub Pages零成本部署，无需服务器维护。同时通过JavaScript实现中英文双语即时切换，不刷新页面。

### 1.5 与现有软件的对比

当前市场上缺乏同类开源Robotaxi远程运营管理平台。Waymo、Tesla、百度Apollo、Pony.ai等运营商均使用自研的闭源系统，且技术细节不对外公开。学术界的相关研究分散在不同论文中，缺乏集成化的工具平台。本软件填补了这一空白，为行业提供首个开放的基础参考实现。

### 2.2 四个核心模块

**模块一：事件数据库（Incident Database）**
采用YAML文件存储每起Robotaxi异常事件，所有事件文件放置在`incidents/`目录下，按年份组织为子目录。每个事件文件遵循统一的JSON Schema约束。

**模块二：场景分类体系（Scenario Taxonomy）**
定义6大类27子场景的分类体系，存储在`taxonomy/`目录。每个子场景有独立编码（如A1、B2），配备严重度和紧急度分级，支持事件的多维分类。

**模块三：三层参考架构（Reference Architecture）**
提出三层决策模型，存储在`architecture/`目录。每层对应特定的异常处置方式、响应时间、人员角色和通信要求。

**模块四：评价基准（Evaluation Benchmarks）**
定义8项KPI及其目标值，存储在`benchmarks/`目录。支持远程运营质量的量化评估。

### 2.3 数据流与控制流

**数据流：**
用户（运营商/研究者/监管者）→ 通过网页浏览或GitHub克隆 → 获取事件数据库/分类体系/架构文档 → 应用到自身运营或研究。

**控制流：**
贡献者发现新事件 → 按YAML Schema填写事件文件 → 提交Pull Request → 维护者审核验证来源链接 → 合并到主分支 → 网站自动更新。

## 第三章 事件数据库模块

### 3.1 YAML Schema设计

事件数据库模块的核心是一个JSON Schema定义文件（`incidents/schema.json`），它约束每起事件的字段结构和允许值。Schema的顶层对象包含以下必填和可选字段：

**必填字段（7个）：**
- `id`: 事件唯一标识，格式为"ROAM-YYYY-NNN"
- `date`: 事件发生日期，ISO 8601格式
- `operator`: 运营商名称
- `location`: 事件发生位置（城市+国家+道路类型）
- `scenario`: 场景分类编码（主要场景+次要场景）
- `severity`: 严重度等级（S0-S4）
- `description`: 事件自然语言描述

**可选字段：**
- `time`: 具体时间
- `vehicle_model`: 车辆型号
- `urgency`: 紧急度等级（U0-U3）
- `impact`: 影响范围（受影响车辆数、持续时间、交通中断等级、伤亡）
- `emergency_response`: 应急响应情况
- `root_cause`: 根因分析
- `systemic_issues`: 系统性问题
- `regulatory_action`: 监管行动
- `sources`: 来源链接（至少1个可验证URL）
- `contributor`: 贡献者
- `last_updated`: 最后更新日期

### 3.2 事件记录字段说明

**严重度分级（severity）：**
- S0 (Near-Miss): 未遂事件，无实际影响
- S1 (Minor): 轻微事件，局部短时交通干扰
- S2 (Moderate): 中等事件，显著交通中断或轻伤
- S3 (Severe): 严重事件，大规模瘫痪或重伤
- S4 (Critical): 致命事件，死亡或大规模系统失效

**紧急度分级（urgency）：**
- U0 (Low): 可延后处理，不影响安全
- U1 (Medium): 30分钟内响应
- U2 (High): 10分钟内响应
- U3 (Immediate): 立即响应，乘客安全受威胁

**交通中断等级（traffic_disruption）：**
- none（无）、minor（轻微）、moderate（中等）、severe（严重）、critical（关键）

### 3.3 数据验证工具

本模块提供Python验证脚本，支持以下功能：
- 检查每个YAML文件是否符合Schema约束
- 验证所有来源URL可访问
- 检查事件ID唯一性
- 统计事件按运营商、年份、场景类型、严重度的分布
- 生成数据质量报告

### 3.4 查询与统计功能

提供Python查询API，支持：
- 按运营商筛选事件
- 按时间范围筛选
- 按场景类型筛选
- 按严重度/紧急度筛选
- 多条件组合查询
- 统计报表生成

当前数据库已收录16起事件：
- Waymo：10起（含2023-2026跨年事件）
- Cruise：2起
- 百度Apollo：3起
- Pony.ai：1起

## 第四章 场景分类体系模块

### 4.1 六大类场景定义

本软件提出的场景分类体系包含6个主类别，每个类别代表一类根本性的异常成因：

**A. 系统性故障 (System-Wide Failure)**
由基础设施、云端或平台级失效导致的多车同时异常。这是最高风险类别，可在几分钟内瘫痪整个车队。

**B. 感知/决策失效 (Perception/Decision Failure)**
由车端感知系统误识别、决策系统输出异常导致的单车异常。

**C. 规划/执行异常 (Planning/Execution Anomaly)**
由路径规划或控制执行错误导致的车辆行为异常。

**D. 车辆硬件故障 (Vehicle Hardware Failure)**
由传感器、动力、制动、转向等硬件部件故障导致的异常。

**E. 外部环境冲突 (External Conflict)**
由外部交通参与者、道路施工、应急车辆、交警指挥等导致的异常。

**F. 乘客端异常 (Passenger-Side Issue)**
由乘客被困、不当行为、医疗紧急等乘客相关事件导致的异常。

### 4.2 二十七子场景详述

每个主类别进一步细分为4-5个子场景，共27个子场景：

**A类（系统性故障）：**
- A1: 云端/网络大面积故障
- A2: OTA推送异常导致批量失效
- A3: 外部基础设施故障（停电、通信基站）
- A4: 服务器端决策系统崩溃

**B类（感知/决策失效）：**
- B1: 复杂路口决策犹豫
- B2: 异常物体误识别
- B3: 极端天气感知退化
- B4: 动态交通参与者异常行为
- B5: GPS/定位漂移

**C类（规划/执行异常）：**
- C1: 路中停车/冻结
- C2: 意外刹车导致追尾
- C3: 危险变道/跨多车道
- C4: 碰撞静态物体
- C5: 低速碰撞行人/骑行者

**D类（车辆硬件故障）：**
- D1: 传感器故障
- D2: 动力系统/电池故障
- D3: 车辆起火
- D4: 制动/转向执行器失效

**E类（外部环境冲突）：**
- E1: 被其他车辆碰撞
- E2: 道路施工/临时封路
- E3: 紧急车辆让行失败
- E4: 交警手势指挥理解失败

**F类（乘客端异常）：**
- F1: 乘客被困车内
- F2: 乘客在危险位置上下车
- F3: 乘客干扰车辆运行
- F4: 乘客紧急医疗事件

### 4.3 严重度S0-S4分级

严重度评估基于事件的实际后果，包括：
- 受影响车辆数量
- 交通中断持续时间
- 伤亡情况
- 系统性影响范围

每个严重度等级有明确的判定阈值，支持事件的客观分级。

### 4.4 紧急度U0-U3分级

紧急度评估基于事件的响应时间敏感性：
- 是否威胁乘客人身安全
- 是否阻碍应急车辆通行
- 是否造成持续扩大的交通影响
- 是否涉及财产损失风险

严重度与紧急度是两个独立的维度，共同决定异常处置的优先级和资源分配。

## 第五章 三层参考架构模块

### 5.1 Layer 1: AI自主响应

**目标覆盖率：** 70%的异常事件
**响应方式：** 无需人工介入，ADS自主处置
**典型场景：** 靠边停车、低速绕障、感知模块重启、乘客语音安抚

**对齐GB标准：** 附录C.2.1.2明确规定"ADS应不依赖远程协助执行DDT（动态驾驶任务）"。

**技术要求：**
- 边缘决策能力（不依赖云端）
- 自诊断能力
- 最小风险条件（MRC）预案库
- 本地感知冗余

### 5.2 Layer 2: AI辅助+人工确认

**目标覆盖率：** 25%的异常事件
**响应方式：** AI生成处置方案（含置信度），远程安全员审核确认或修改
**典型场景：** 复杂路口脱困、交警手势解读、多车冲突裁决、报警决策

**对齐GB标准：** 附录C.2.3规定"ADS应能够及时接收远程协助指令"，C.2.1.4规定"ADS应基于行驶环境安全性响应远程协助指令"。

**技术要求：**
- 端到端延迟<500ms
- 操作员工作台三段式UI
- AI预计算替代方案
- 决策审计日志

### 5.3 Layer 3: 远程驾驶/线下派人

**目标覆盖率：** 5%的异常事件
**响应方式：** 远程直接接管控制（≤10km/h）或派遣现场救援团队
**典型场景：** 车辆硬件故障、批量系统瘫痪、事故现场处理、车辆起火

**对齐框架：** 德国StVFernLV远程驾驶法规、Tesla Teleops模式。

**技术要求：**
- 端到端延迟<150ms
- 物理层专用通信链路（如DJI OcuSync-style图传）
- 力反馈控制器
- 物理急停按钮

### 5.4 场景-处置映射表

本模块提供完整的27子场景到三层架构的映射表，每行记录：
- 场景编码
- 主推荐层级
- 升级触发条件
- 典型处置流程
- 响应时间要求

此映射表是运营商快速设计自己远程运营系统的核心参考。

## 第六章 评价基准模块

### 6.1 八项KPI定义

本软件定义以下8项关键性能指标：

**KPI-1: MTTR（平均处置时间）**
定义：从异常发生到恢复正常的平均时间
目标值：AI自主处置<5分钟，人工确认<15分钟

**KPI-2: AI自主处置率**
定义：AI无需人工即可解决的异常占比
目标值：>70%

**KPI-3: 误升级率**
定义：AI本可自主处置但升级到人工的比例
目标值：<10%

**KPI-4: 漏升级率**
定义：AI未升级但事后证明应升级的比例
目标值：<1%（安全关键指标）

**KPI-5: 乘客被困时长**
定义：从异常发生到乘客安全离开的时间
目标值：<10分钟

**KPI-6: 交通恢复时间**
定义：从车辆停滞到道路恢复通行的时间
目标值：<20分钟

**KPI-7: 人车比**
定义：远程安全员与管理车辆的比例
目标值：1:80 - 1:100（配合AI辅助后）

## 第七章 双语网站模块

### 7.1 中英文切换机制

本软件的Web界面支持中英文双语无刷新切换。实现机制：

- HTML中所有双语内容使用`<span class="cn">中文</span><span class="en">English</span>`模式
- CSS通过body类名`E`（英文模式）控制显示/隐藏
- JavaScript函数`L(lang, btn)`切换body类名并更新按钮状态
- 默认为中文，右上角提供"中文/EN"切换按钮

### 7.2 响应式布局

网站采用响应式设计，适配桌面、平板、手机屏幕：
- 桌面：1320px最大宽度
- 平板：768px断点，网格降级
- 手机：640px断点，所有卡片单列

使用CSS `@media` 查询和 `vw` 单位实现流式字体大小。

### 7.3 GitHub Pages部署

网站通过GitHub Pages零成本部署：
- 源文件位于`docs/`目录
- Jekyll自动构建
- HTTPS默认启用
- 自定义域名支持（通过CNAME文件）

### 7.4 字体设计

采用Google Fonts：
- Playfair Display（衬线字体）用于标题
- Inter（无衬线字体）用于正文
- 系统字体回退（PingFang SC、Microsoft YaHei）

```bash
git clone https://github.com/AutoZYX/ROAM.git

```

### 8.3 事件数据查询

使用Python脚本查询事件：

```python
import yaml
import glob

# 加载所有事件
events = []
for f in glob.glob('incidents/*/ROAM-*.yaml'):
    with open(f) as fp:
        events.append(yaml.safe_load(fp))

# 查询：严重度S3以上的事件
severe = [e for e in events if e['severity'] in ['S3', 'S4']]
print(f"严重事件：{len(severe)}起")
```

### 8.4 贡献新事件

1. Fork仓库
2. 在`incidents/YYYY/`目录创建新YAML文件
3. 按Schema填写字段
4. 确保所有来源URL可访问
5. 提交Pull Request
