## 第一章 软件概述

### 1.1 开发背景

随着L4级Robotaxi在中美欧等国家快速商业化部署（Waymo 3000+辆、百度Apollo 2025Q4完成340万单），保险行业面临一个根本性挑战：**如何为Robotaxi事故定价？**

传统汽车保险的定价基于驾驶员属性（年龄、驾龄、历史事故）和车辆属性（品牌、型号）。但Robotaxi的"驾驶员"是AI系统，且各运营商使用不同的ADS技术栈、在不同城市环境、面对不同交通模式。现有保险精算模型无法直接迁移。

当前行业的尝试（如Waymo与瑞士再保险合作）普遍面临共性挑战：**缺乏标准化的事件数据和风险评估框架**。各运营商的事故报告格式不统一，严重度和可预防性的判断标准缺失，保险公司无法建立可比较的精算模型。

### 1.2 软件目的

本软件的目的是：**为Robotaxi保险提供数据驱动的可预防性量化和差异化定价能力**。

### 1.3 核心创新

本软件的核心创新是将**事故数据**（事故分子）与**自然驾驶行为基线**（正常行为分母）通过"可预防性指数（Preventability Index, PI）"进行融合：

- **分子：** ROAM事件数据库（含各运营商Robotaxi事故记录）
- **分母：** DRIVEResearch航测自然驾驶数据集（750h+飞行，1050万+轨迹）
- **指数：** PI = f(事故行为参数, 基线百分位分布)

通过PI可以回答一个关键问题：**该起事故是"人类也无法避免"（不可预防）还是"系统本应避免"（应可预防）？**

- PI-A（不可预防）：人类驾驶员在相同场景下也会发生事故，属于合理自然风险
- PI-B（边界可预防）：少数优秀人类驾驶员可避免，边界情形
- PI-C（应可预防）：大多数人类驾驶员可避免，属于ADS系统缺陷

### 1.4 与现有软件的对比

| 维度 | 传统车险精算软件 | 本软件 |
|------|----------------|--------|
| 定价基础 | 驾驶员属性+历史事故 | ADS能力+自然驾驶基线 |
| 数据来源 | 保险公司理赔数据 | 事故数据库+航测NDD |
| 可预防性判断 | 人工审核（主观） | PI指数量化（客观） |
| 差异化维度 | 车辆品牌+驾驶员 | 运营商+城市+场景 |
| 输出 | 单一保费 | 保费+评级+分级依据 |

## 第三章 事件数据解析模块

### 3.1 模块功能

读取ROAM事件数据库YAML文件，提取精算所需字段。

### 3.2 核心字段提取

从每起事件中提取以下参数：
- **场景类型**: scenario.primary（A1-F4）
- **严重度**: severity（S0-S4）
- **紧急度**: urgency（U0-U3）
- **运营商**: operator
- **地点**: location.city, location.country, location.road_type
- **车辆行为特征**: 从description中NLP提取速度、加速度、TTC估算值
- **后果**: impact字段中的vehicles_affected, duration, injuries, fatalities

### 3.3 参数化输出

事件向量化表示：
```python
IncidentVector = {
    'id': str,
    'operator': str,
    'scenario_code': str,
    'severity_level': int,  # 0-4
    'urgency_level': int,  # 0-3
    'speed_estimate_kmh': float,
    'ttc_estimate_s': float,
    'location_key': str,  # 城市+道路类型
    'outcome_score': float  # 综合伤亡+中断
}
```

## 第四章 自然驾驶基线集成模块

### 4.1 数据源：DRIVEResearch航测NDD

本模块对接驭研科技独有的航测自然驾驶数据集：
- **飞行时数：** 750小时+（持续扩展）
- **轨迹数量：** 1050万+
- **覆盖城市：** 长春、深圳、武汉、北京等10+城市
- **场景类型：** 高速公路、城市主干道、环岛、匝道、停车场等
- **数据频率：** 25Hz位置+速度+加速度

### 4.2 场景匹配算法

按以下维度检索基线：
- **场景类型**：匹配到相同ROAM场景编码
- **环境条件**：道路类型、天气、时段
- **速度区间**：20-40, 40-60, 60-80, 80-100, 100-120 km/h

### 4.3 百分位分布计算

对匹配到的基线样本，计算以下参数的百分位分布：
- TTC（Time-To-Collision）
- THW（Time Headway）
- 加速度
- 横向加速度
- 车速偏差

输出 P5/P25/P50/P75/P95 百分位值。

## 第五章 可预防性指数(PI)计算模块

### 5.1 PI三级分类定义

| PI等级 | 含义 | 事故行为位置 | 保费系数 |
|--------|------|------------|---------|
| **PI-A** | 不可预防 | 基线P5以下（极端不利） | 1.0 |
| **PI-B** | 边界可预防 | 基线P5-P25之间 | 1.5 |
| **PI-C** | 应可预防 | 基线P25以上 | 3.0 |

### 5.2 算法流程

```
1. 从事件向量读取行为参数（速度、TTC、加速度等）
2. 从基线库读取匹配场景的百分位分布
3. 计算事故行为在基线分布中的百分位位置
4. 根据百分位位置判定PI等级
5. 输出PI等级+置信度
```

### 5.3 置信度评估

基于样本量、匹配度、参数完整性三维评估：
- 基线样本>1000: confidence=0.9
- 基线样本100-1000: confidence=0.7
- 基线样本<100: confidence=0.4

## 第六章 事故暴露率估算模块

### 6.1 暴露率定义

**事故暴露率** = 特定场景事故频次 / 基线场景暴露量

单位转换：
- 事故/百万公里
- 事故/小时运营
- 事故/车·年

### 6.2 与GB标准对齐

中国GB《智能网联汽车 自动驾驶系统安全要求》附录D残余风险接受准则：
- 碰撞事故率 < 10⁻⁴/h
- 轻伤事故率 < 10⁻⁵/h
- 重伤事故率 < 10⁻⁶/h
- 致命事故率 < 10⁻⁷/h

本软件计算每个运营商的实际事故率，与GB准则对比输出合规/风险报告。

### 6.3 计算示例

```python
# 运营商A的PI-C类事故率
pi_c_incidents = 12  # 近一年PI-C事故数
total_operating_hours = 50000  # 运营总小时数
pi_c_rate = pi_c_incidents / total_operating_hours  # = 2.4×10⁻⁴/h

# 对比GB准则10⁻⁴/h，超标2.4倍 → 需保费加成
```

## 第七章 风险定价模型模块

### 7.1 保费定价公式

```
最终保费 = 基础保费 × PI系数 × 运营商评级系数 × 城市环境系数
```

### 7.2 运营商评级（1-5星）

基于运营商所有事件的PI分布：
- ★★★★★（5星）：PI-C占比<5%，无致命事故
- ★★★★（4星）：PI-C占比5-10%，无致命事故
- ★★★（3星）：PI-C占比10-20%
- ★★（2星）：PI-C占比20-40%
- ★（1星）：PI-C占比>40% 或有致命事故

### 7.3 城市环境系数

基于城市复杂度：
- 一线城市复杂环境（北京、上海、深圳）：1.2
- 二线城市常规环境：1.0
- 郊区简单环境：0.8

## 第十章 技术规格

### 10.1 性能指标

- 单事件PI计算：<100ms
- 批量报告生成（100起事件）：<30秒
- API响应时间：<200ms

### 10.2 算法复杂度

- 基线匹配：O(n log n)，n=基线样本数
- PI计算：O(1)
- 运营商评级：O(k)，k=运营商事件数

### 10.3 可扩展性

- 支持新增运营商（无需修改代码）
- 支持新增城市（配置文件）
- 支持新增场景类型（配合ROAM更新）
